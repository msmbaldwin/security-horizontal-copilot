## Content Requirements

### Security Domains (Include Only Applicable)
Order these sections according to Service Classification rules above:

- **Service-Specific Security** — Unique security controls, architecture patterns, anti-patterns, AND platform security context (what the service secures automatically vs. customer responsibility)

- **Backup & Recovery** — Data protection, business continuity, automatic redundancy

### Required Content Types

#### Positive Recommendations (What TO Do)
Standard best practice recommendations with action verbs.

#### Anti-Patterns (What NOT To Do) - Include When Documentation Supports
Include guidance on inappropriate use when official documentation describes anti-patterns, misuse scenarios, or inappropriate use cases. Do not fabricate anti-patterns—only include those supported by Microsoft documentation:

- What data types should NOT be stored in this service
- What the service should NOT be used for
- Common misconfigurations to avoid
- Architectural anti-patterns

Format anti-patterns as:
```markdown
- **Do not use <Service> for <inappropriate use>**: <Explanation of why this is wrong and what to use instead>. See [Alternative approach](/azure/alternative).
```

#### Conditional Recommendations
When recommendations have prerequisites or exceptions, format as sub-bullets:
```markdown
- **Primary recommendation**: Explanation. See [Link](/path).
    - Some scenarios require <exception>. In such cases, <alternative approach>. See [Exception guidance](/path).
```

### Quality Standards
- Every recommendation must link to authoritative Microsoft documentation
- Use site-relative paths for all links (e.g., `/azure/service/feature`)
- No placeholder content or generic boilerplate
- Follow `template.txt` structure and `guide.txt` writing guidelines
- Include `ms.custom: horz-security` in YAML frontmatter
- **Do NOT use `[!IMPORTANT]`, `[!NOTE]`, or other callout formatting**—integrate all critical information directly into the recommendation text
- **Only include actionable recommendations that represent customer choices**:
  - If a security feature is enforced by the service with no option to disable it, don't present it as a best practice (e.g., "Use TLS 1.2" when the service only supports TLS 1.2+)
  - If a feature is built into the architecture with no configuration required, mention it as context rather than a recommendation (e.g., built-in high availability)
  - Non-actionable items like "Understand the shared responsibility model" should be removed or reframed as actionable guidance
  - Focus on configurations customers can actually choose to enable, disable, or configure
- Include anti-patterns only when supported by official documentation
- Section ordering MUST follow standard order from guide.txt
